# v0.2.0 — SRS Loop Completion
Version: 0.2.0 | Status: Built, uncommitted | Date: 2026-03-17

> Closes the spaced repetition feedback loop: per-word results are now tracked during a quiz and
> written back to UserWordProgress after each session. Exercises are also fetched in SRS priority
> order. Includes a pre-quiz start screen for session configuration.

---

## The Problem

The initial MVP had a broken SRS loop:

1. **Exercise delivery ignored SRS state.** `GET /api/exercises` fetched words ordered by `difficulty: asc` from the `Exercise` table, which had no entries — it fell through to a fallback that returned random words. `UserWordProgress` (ease factor, nextReviewAt, status) was never consulted.

2. **Per-word results were never persisted.** `ExerciseContainer` tracked `correct` and `total` counts but threw away which specific words were answered correctly or incorrectly. `POST /api/sessions` received no per-word data, so `UserWordProgress` was never updated after a session.

3. **Quiz started loading immediately.** Exercises fetched as soon as the page loaded, before the user set their desired session length. The Pinyin toggle was buried in the quiz header.

The net effect: the SM-2 algorithm existed in `lib/srs/sm2.ts` but was wired to nothing. Every session was identical — the same random words, no prioritization of due reviews, no memory of past performance.

---

## Validation Stories

> **Story 1 — "SRS remembers what I got wrong"**
>
> Practice 10 words. Get 3 wrong. Finish the session.
> The next time you open a quiz at the same level, those 3 words appear first.
> Words you got right are scheduled further out and appear later in the queue.

> **Story 2 — "Session setup before I start"**
>
> Navigate to `/practice/quiz?level=2`. Instead of immediately seeing a loading spinner,
> you see a setup screen: choose 5, 10, 20, 30, 50 words (or drag a slider up to 150).
> Toggle Pinyin on/off. Click Start. Only then does the quiz load.

> **Story 3 — "New user gets a clean random start"**
>
> A logged-out visitor navigates to `/practice/quiz?level=1`.
> They get a shuffled set of words with no SRS ordering (no progress to order by).
> The experience still works — just without personalization.

## What Was Built

### 1. Per-word result tracking (ExerciseContainer)

`components/exercises/ExerciseContainer.tsx`

- Added `wordResults: WordResult[]` state (`{ wordId: string; isCorrect: boolean }[]`)
- Each `handleAnswer` call appends to `wordResults` if `current.wordId` is set
- `onComplete` callback signature extended to include `wordResults`
- Fixed a latent bug: `onComplete` was calculating `correct` incorrectly (adding 0 regardless)

### 2. SRS-aware exercise ordering (exercises API)

`app/api/exercises/route.ts` — full rewrite

- Removed the `Exercise` table lookup (table is empty; the fallback was the real path)
- Now queries `HskWord` directly for the level, plus `UserWordProgress` for authenticated users
- Ordering priority for authenticated users:
  1. **Due for review** — `status !== 'mastered'` AND `nextReviewAt <= now` (shuffled)
  2. **New words** — never seen (shuffled)
  3. **In-progress, not yet due** — `status !== 'mastered'` AND `nextReviewAt > now` (shuffled)
  4. **Mastered** — reinforcement buffer (shuffled)
- Falls back to random shuffle if not enough words fill the requested count
- Unauthenticated users get a random shuffle (no `UserWordProgress` to consult)
- Count capped at 200

### 3. SRS progress persistence (sessions API)

`app/api/sessions/route.ts`

- Accepts `wordResults: { wordId: string; isCorrect: boolean }[]` in POST body
- For each word result, updates or creates `UserWordProgress`:
  - **Correct:** advances status (`new → learning → reviewing → mastered`) and increases interval
  - **Incorrect:** regresses status (`mastered/reviewing → learning`), resets interval to 1 day, reduces ease factor by 0.15 (floor 1.3)
  - **First time seen:** creates a new progress record
- SM-2 status transitions:
  | From | Correct → | Interval |
  |------|-----------|----------|
  | `new` | `learning` | 1 day |
  | `learning` | `reviewing` | 3 days |
  | `reviewing` | `mastered` | 7 days |
  | `mastered` | `mastered` | `interval × easeFactor` |
  | any | (incorrect) | 1 day |

### 4. Pre-quiz start screen (quiz page)

`app/(app)/practice/quiz/page.tsx`

- Added `started` state — exercises only fetch after user clicks Start
- Pre-quiz screen shows:
  - Heading: "HSK {level} Practice" + subtitle
  - Word count preset buttons: 5, 10, 20, 30, 50
  - Range slider: 1–150 with live count display
  - Pinyin toggle (moved here from in-quiz header)
  - Start button
- During quiz, Pinyin toggle replaced with a static badge (read-only indicator)
- `handleRetry` simplified — sets `started = false` instead of manually re-fetching

## Verification

### Known Gaps / Follow-up

- **SM-2 algorithm not fully used:** The SRS transitions in `sessions/route.ts` are a simplified version (hardcoded intervals: 1/3/7 days). The full SM-2 algorithm in `lib/srs/sm2.ts` (ease factor calculation, quality grades 0–5) is not yet called. Should be wired in a follow-up.
- **`Exercise` table unused:** The schema has an `Exercise` model for pre-authored exercises (with distractors, grammar exercises, etc.). The current flow generates everything dynamically from `HskWord`. The Exercise table should either be populated or removed in a future cleanup.
- **No error state for empty word list:** If a level has no words, the quiz silently shows nothing.
